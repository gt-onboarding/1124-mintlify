Wenn Sie derzeit einzelne `MDX`-Seiten für Ihre API-Endpoints verwenden, können Sie auf das automatische Generieren von Seiten aus Ihrer OpenAPI-Spezifikation umsteigen und dabei die Anpassbarkeit einzelner Seiten beibehalten. Dadurch können Sie die Anzahl der zu verwaltenden Dateien reduzieren und die Konsistenz Ihrer API-Dokumentation verbessern.

Sie können Metadaten und Inhalte für jeden Endpoint in Ihrer OpenAPI-Spezifikation definieren und die Endpoints in Ihrer Navigation an den gewünschten Stellen organisieren.

<div id="cli-migration">
  ## CLI-Migration
</div>

Der Befehl `mint migrate-mdx` ist die empfohlene Methode, um von MDX-Endpoint-Seiten zu automatisch generierten Seiten zu migrieren.

Dieser Befehl:

* Analysiert die `docs.json`-Navigationsstruktur.
* Identifiziert MDX-Seiten, die OpenAPI-Endpoint-Seiten erzeugen.
* Extrahiert Inhalte aus MDX-Dateien und verschiebt sie in die `x-mint`-Erweiterung deiner OpenAPI-Spezifikation.
* Aktualisiert deine `docs.json`, sodass direkt auf die OpenAPI-Endpoints statt auf MDX-Dateien verwiesen wird.
* Löscht die ursprünglichen MDX-Endpoint-Dateien.

<Info>
  Wenn für einen Endpoint bereits `x-mint` definiert ist und es zudem eine MDX-Seite mit Inhalt für diesen Endpoint gibt, überschreibt der MDX-Inhalt die vorhandenen `x-mint`-Einstellungen.

  Wenn es mehrere MDX-Seiten für denselben Endpoint mit unterschiedlichem Inhalt gibt, verwendet das Skript den Inhalt der Seite, die in deiner `docs.json` zuletzt erscheint.

  Das Migrationstool unterstützt keine Vorschau der Änderungen, bevor sie angewendet werden.
</Info>

<Steps>
  <Step title="Bereite deine OpenAPI-Spezifikation vor.">
    Stelle sicher, dass deine OpenAPI-Spezifikation gültig ist und alle Endpoints enthält, die du dokumentieren möchtest.

    Alle MDX-Seiten, die du migrieren möchtest, müssen das `openapi:`-Frontmatter mit Verweis auf einen Endpoint enthalten.

    <Tip>
      Validiere deine OpenAPI-Datei mit dem [Swagger Editor](https://editor.swagger.io/) oder der [Mint CLI](https://www.npmjs.com/package/mint).
    </Tip>
  </Step>

  <Step title="Installiere die Mint CLI">
    Installiere bei Bedarf die [Mint CLI](/de/installation) oder führe ein Update durch.
  </Step>

  <Step title="Führe den Migrationsbefehl aus.">
    ```bash
    mint migrate-mdx
    ```
  </Step>
</Steps>

<div id="manual-migration-steps">
  ## Manuelle Migrationsschritte
</div>

<Steps>
  <Step title="Bereiten Sie Ihre OpenAPI-Spezifikation vor.">
    Stellen Sie sicher, dass Ihre OpenAPI-Spezifikation gültig ist und alle Endpoints enthält, die Sie dokumentieren möchten.

    Für Endpoints, bei denen Sie Metadaten oder Inhalte anpassen möchten, fügen Sie dem Endpoint die Erweiterung `x-mint` hinzu. Weitere Details finden Sie unter [x-mint-Erweiterung](/de/api-playground/openapi-setup#x-mint-extension).

    Für Endpoints, die Sie aus Ihrer Dokumentation ausschließen möchten, fügen Sie dem Endpoint die Erweiterung `x-hidden` hinzu.

    <Info>
      Validieren Sie Ihre OpenAPI-Datei mit dem [Swagger Editor](https://editor.swagger.io/) oder der [Mint CLI](https://www.npmjs.com/package/mint).
    </Info>
  </Step>

  <Step title="Aktualisieren Sie Ihre Navigationsstruktur.">
    Ersetzen Sie `MDX`-Seitenverweise in Ihrer `docs.json` durch OpenAPI-Endpoints.

    ```json
    "navigation": {
      "groups": [
        {
          "group": "API Reference",
          "openapi": "/path/to/openapi.json",
          "pages": [
            "overview",
            "authentication",
            "introduction",
            "GET /health",
            "quickstart", 
            "POST /users",
            "GET /users/{id}",
            "advanced-features"
          ]
        }
      ]
    }
    ```
  </Step>

  <Step title="Entfernen Sie alte MDX-Dateien.">
    Nachdem Sie überprüft haben, dass Ihre neue Navigation korrekt funktioniert, entfernen Sie die `MDX`-Endpoint-Dateien, die Sie nicht mehr benötigen.
  </Step>
</Steps>

<div id="navigation-patterns">
  ## Navigationsmuster
</div>

Sie können festlegen, wie Ihre API-Dokumentation in der Navigation angezeigt wird.

<div id="mixed-content-navigation">
  ### Gemischte Content-Navigation
</div>

Kombinieren Sie automatisch generierte API-Seiten mit anderen Seiten:

```json
"navigation": {
  "groups": [
    {
      "group": "API-Referenz",
      "openapi": "openapi.json",
      "pages": [
        "api/overview",
        "GET /users",
        "POST /users", 
        "api/authentifizierung"
      ]
    }
  ]
}
```

<div id="multiple-api-versions">
  ### Mehrere API-Versionen
</div>

Organisieren Sie verschiedene API-Versionen mit Tabs oder Gruppen:

```json
"navigation": {
  "tabs": [
    {
      "tab": "API v1",
      "openapi": "specs/v1.json"
    },
    {
      "tab": "API v2", 
      "openapi": "specs/v2.json"
    }
  ]
}
```

<div id="when-to-use-individual-mdx-pages">
  ## Wann einzelne `MDX`-Seiten sinnvoll sind
</div>

Behalten Sie einzelne `MDX`-Seiten bei, wenn Sie Folgendes benötigen:

* Umfangreiche, individuell gestaltete Inhalte pro Endpoint, z. B. React-Komponenten oder ausführliche Beispiele.
* Einzigartige Seitenlayouts.
* Experimentelle Dokumentationsansätze für bestimmte Endpoints.

Für die meisten Anwendungsfälle bietet die OpenAPI-Navigation bessere Wartbarkeit und Konsistenz.
