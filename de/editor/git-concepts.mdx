Git ist ein Versionskontrollsystem, mit dem Sie Änderungen an Ihrer Dokumentation nachverfolgen und im Team zusammenarbeiten können. Mit Git können Sie für jede Datei in Ihrem Projekt sehen, was sich geändert hat, wann und warum. Außerdem können Sie bei Bedarf problemlos auf frühere Versionen von Dateien zurücksetzen.

Der Web-Editor führt einige Git-Operationen im Hintergrund aus. Wenn Sie diese Konzepte verstehen, arbeiten Sie effektiver mit dem Web-Editor und können besser mit Teammitgliedern zusammenarbeiten, die in ihren lokalen Entwicklungsumgebungen arbeiten.

<div id="core-git-concepts">
  ## Zentrale Git-Konzepte
</div>

<AccordionGroup>
  <Accordion title="Repository">
    Die Quelle Ihrer Dokumentation, in der alle Dateien und deren Historie gespeichert sind. Der Web-Editor verbindet sich mit Ihrem Repository, um auf Inhalte zuzugreifen und sie zu bearbeiten.
  </Accordion>

  <Accordion title="Commit">
    Ein gespeichertes Abbild Ihrer Änderungen zu einem bestimmten Zeitpunkt. Jeder Commit enthält eine Nachricht, die beschreibt, was geändert wurde, und erstellt einen dauerhaften Eintrag in der Projektgeschichte.
  </Accordion>

  <Accordion title="Branch">
    Ein Branch verweist auf einen bestimmten Commit in Ihrem Repository. Ihre Live-Dokumentation wird aus einem Deployment-Branch erstellt. Sie können beliebig viele weitere Branches haben, deren Änderungen noch nicht in Ihrer Live-Dokumentation veröffentlicht sind. Wenn Sie die Änderungen aus einem Branch in Ihre Live-Dokumentation übernehmen möchten, können Sie den Branch über einen Pull Request in Ihren Deployment-Branch mergen.

    Verwenden Sie Branches, um an Änderungen zu arbeiten, ohne Ihre Live-Dokumentation zu beeinträchtigen, neue Funktionen sicher zu erproben und Reviews vor der Veröffentlichung zu erhalten.
  </Accordion>

  <Accordion title="Deployment branch">
    Der primäre Branch Ihres Projekts, aus dem der Inhalt Ihrer Live-Dokumentation erstellt wird. Änderungen an diesem Branch werden automatisch auf Ihrer Dokumentationsseite veröffentlicht. Häufig „main“ genannt, aber Sie können jeden Branch als Ihren Deployment-Branch festlegen.
  </Accordion>

  <Accordion title="Pull request">
    Eine Möglichkeit, das Mergen Ihrer Änderungen von einem Branch in Ihre Live-Dokumentation vorzuschlagen. Ermöglicht Review und Diskussion, bevor Änderungen live gehen. Üblicherweise PR genannt, in GitLab auch Merge Request.
  </Accordion>

  <Accordion title="Diff">
    Ein Diff (oder Unterschied) zeigt die Änderungen zwischen zwei Versionen einer Datei. Beim Review von Pull Requests heben Diffs hervor, was hinzugefügt, entfernt oder geändert wurde, sodass leicht erkennbar ist, was sich geändert hat.
  </Accordion>
</AccordionGroup>

<div id="how-the-web-editor-uses-git">
  ## Wie der Web-Editor Git verwendet
</div>

Der Web-Editor verbindet sich über die [GitHub-App](/de/settings/github) oder die [GitLab-Integration](/de/settings/gitlab) mit deinem Git-Repository und automatisiert gängige Git-Vorgänge.

Wenn du:

* **Eine Datei öffnest**: Der Editor lädt die neueste Version aus deinem Repository, sodass du stets mit aktuellem Inhalt arbeitest.
* **Änderungen vornimmst**: Der Editor verfolgt deine Änderungen als Entwurf, der zu einem Commit werden kann, sobald du bereit bist, deine Arbeit zu speichern.
* **Änderungen speicherst**: Der Editor erstellt einen Commit mit deinen Änderungen und bewahrt deine Arbeit in der Projekt-Historie auf.
* **Einen Branch erstellst**: Der Editor erstellt einen neuen Branch in deinem Repository, den alle mit Repository-Zugriff nutzen können, um zusammenzuarbeiten und Änderungen zu prüfen.
* **Auf deinem Deployment-Branch veröffentlichst**: Der Editor committed und pusht direkt auf deinen Deployment-Branch, wodurch deine Änderungen sofort veröffentlicht werden.
* **Auf anderen Branches veröffentlichst**: Der Editor erstellt einen Pull Request, sodass du Feedback von anderen erhalten kannst, bevor du deine Änderungen in deinen Deployment-Branch mergest.

<div id="git-best-practices">
  ## Git-Best Practices
</div>

Jedes Team entwickelt eigene Workflows und Präferenzen, aber hier sind einige allgemeine Best Practices für den Einstieg.

* **Aussagekräftige Commit-Nachrichten schreiben**: Formuliere konkret, was sich geändert hat, und verwende aktive Sprache.
* **Aussagekräftige Branch-Namen verwenden**: Branch-Namen sollten die vorgenommenen Arbeiten beschreiben und für alle nachvollziehbar sein, die die Branches in deinem Repository ansehen.
* **Branches fokussiert halten**: Beschränke die Änderungen in einem Branch auf eine konkrete Aufgabe oder ein spezifisches Projekt.
* **Branches nach dem Mergen löschen**: Lösche Branches, wenn du sie nicht mehr benötigst, um dein Repository übersichtlich zu halten.
